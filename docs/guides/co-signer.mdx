---
title: The co-signer
description: "Run The Machine server-side: the enclave that authorizes every spend and vetoes bad ones."
---

The co-signer holds the key that authorizes every spend. It never runs in the
browser, so the key is never exposed. On each spend request it validates the
request against the account's on-chain policy and the poisoning firewall, then
returns a signature or a **veto**. A spend needs only this signature; the funds
are hard-locked to the account's `target` and cap, so the co-signer can only push
the payment the payer set up, never steal or redirect. Fail-closed: if the
firewall's data source is unreachable, it withholds the signature.

## Server endpoint

Expose two routes on the same origin as your app:

- `GET /api/cosign` returns the co-signer's public address, which the browser
  locks into each new account.
- `POST /api/cosign` authorizes a spend and returns a signature or a veto.

```ts
// server-only — the key stays here
import { cosign, cosignerAddress } from '@ctrl-arcz/demo-kit/cosign';

const COSIGNER_PK = process.env.COSIGNER_PK; // never a VITE_/public var

// GET
res.json({ address: cosignerAddress(COSIGNER_PK) });

// POST
const result = await cosign({ privateKey: COSIGNER_PK, body: req.body });
res.json(result); // { approved: true, signature } | { approved: false, reason, riskReasons }
```

`cosign` reads the account's real policy from chain (never trusting what the
browser claims), runs a `LocalCoSigner` backed by the SDK poisoning `check()`, and
signs the exact EIP-712 `Spend` struct the contract will verify.

## The policy the co-signer enforces

Even though the contract re-checks these, the co-signer refuses to put its name on
a request that violates them:

<AccordionGroup>
  <Accordion title="Target lock" icon="crosshairs">
    The co-signer reads the account's real `target` from chain and signs only for
    that address; it never trusts a target the browser supplies. The browser hands
    it only `{account, owner, amount, action}`. A script that swaps the destination
    cannot get a signature, and the contract's `target` is locked anyway.
  </Accordion>
  <Accordion title="Amount and window" icon="gauge">
    The amount must be within the remaining cumulative cap, and the account must
    not have expired.
  </Accordion>
  <Accordion title="Risk firewall" icon="shield-halved">
    Runs `check(owner, target)`. Vetoes on a `block` verdict (drainer, lookalike,
    zero-value bait, or unavailable data). Set `vetoOn: 'warning'` to also stop
    new/fresh recipients.
  </Accordion>
</AccordionGroup>

## In-process co-signer (tests and local)

For tests or a single-process setup, use `LocalCoSigner` directly with a pluggable
risk source:

```ts
import { LocalCoSigner } from '@ctrl-arcz/sdk';

const machine = new LocalCoSigner(COSIGNER_PK, {
  riskCheck: async (owner, target) => ({ level: 'safe', complete: true }),
  vetoOn: 'block',
});

const auth = await machine.authorize(request); // { approved, signature } | { approved: false, reason }
```

<Warning>
The co-signer is a trust point. If it is unreachable, funds are never stranded:
after `expiry`, anyone who knows the vault address can call `sweepExpired(vault)`
to return the account home to the vault. No co-signer signature and no owner
signature are needed, only knowledge of the vault (the preimage of the stored
`vaultHash`). Design your key management around that escape hatch.
</Warning>
