## SpendPolicyFactory + SpendPolicyAccount

The payer-side shield. The factory clones a policy-bound account per intent; the
account enforces the policy and gates every spend on the co-signer.

| Function | Purpose |
| --- | --- |
| `createAccount(ownerHash, userSalt, params)` | Clone + init a disposable account; salt bound to `ownerHash = keccak256(abi.encode(owner))`, not the owner address. Anyone (a relayer) may submit it. |
| `predictAddress(ownerHash, userSalt)` | The deterministic address, so you can fund it first. |
| `pay(amount, cosignerSig)` | PUSH: spend to the locked target, authorized by the co-signer alone. |
| `pull(amount, cosignerSig)` | PULL: co-signer-gated recurring spend; enforces `perPullMax`, `interval`, the cumulative cap and expiry. |
| `sweepToVault(vault)` | Return the balance to the vault. Gated by knowledge of the vault (its hash must match `vaultHash`), no signature. |
| `sweepExpired(vault)` | After expiry, anyone who knows the vault returns the account home. Liveness escape hatch. |

A spend needs only the co-signer's signature. The funds are hard-locked to the
pre-set `target` and capped by `maxAmount`, so the co-signer can only push the
payment the payer set up, never steal or redirect. The payer signs nothing blind;
their only on-chain action is a normal USDC transfer to fund the account.

`init` takes `(token, cosigner, vaultHash, target, maxAmount, perPullMax, expiry,
interval, mode)`. There is no `owner` and no `vault` stored: the owner is bound
only through the salt (`ownerHash`) and the return address only through the
commitment `vaultHash = keccak256(abi.encode(vault))`. The readable getters are
`token`, `cosigner`, `target`, `vaultHash`, `maxAmount`, `perPullMax`, `expiry`,
`interval`, `mode`, `spent` and `nonce`; there is no `owner()` or `vault()` getter
at all, so a merchant paid from the account cannot read who funded it.

Signatures are EIP-712 typed data over `Spend(address target,uint256 amount,uint256
nonce,uint8 action)`, where `action` is 0 for pay and 1 for pull so a pay
authorization can never be replayed as a pull. The domain binds the account and
chain, so a signature cannot be replayed across accounts or chains. Use
`spendTypedData` / `spendDigest` in the SDK to build the exact bytes, or read the
account's `spendDigest(amount, action)` getter.

## Vault

Holds the owner's USDC, funds accounts, receives sweeps. Owner-only outflows; the
`_moveOut` seam is where the APS confidential transfer will slot in.

<Note>
On Arc a USDC ERC-20 transfer moves the recipient's native balance, so the account
and vault implement `receive() payable`. Without it, funding a contract on Arc
would revert; a plain EVM never surfaces this.
</Note>
